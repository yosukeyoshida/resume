### iemo株式会社 (2014.08 - 2017.06)
```
住まい/インテリアのメディアでバックエンドや検索, 機械学習を活用した開発に従事
```
#### 検索基盤の整備
##### 課題/背景
* 全文検索にGroongaが使われていたが仕様を詳しく把握しているエンジニアがいなかった為チューニングや機能追加ができていなかった
* 単純なN-gramによる検索だったので検索結果が雑多になりユーザの求める記事に到達することが困難であった
* インデックスの更新がバッチ処理で行われており記事が更新されてから検索に反映されるまでに時間がかかっていた

##### 実績
* 検索エンジンをGroongaからElasticsearchに移行し形態素解析を用いたトークナイザの追加, タイトル, 本文, タグ等のウェイトの調整を行い検索後の離脱率の改善 (25.28% -> 15.40%) 
* 形態素解析の辞書やシノニムの整備と継続的に誰でもメンテナンスができるように辞書の管理画面を用意
* 記事更新時に非同期でElasticsearchのインデックスを更新することで即座に検索に反映
* コンテンツベースのレコメンデーションを関連記事に活用し、更に表示する関連記事をバンディットアルゴリズムを用いてCTRの最大化, 回遊率の改善

##### 技術
* Ruby on Rails
* MySQL
* Elasticsearch
* SQS
* Redis


#### サイト構造のリニューアル
##### 課題/背景
* サービス開始から1年以上経過し、記事が雑多になり既存のカテゴリ (インテリア, 家具 etc) の枠組みでは対応できなくなってきていた。カテゴリ内の統一性の無さによるユーザビリティの低下 / サイトの世界観の破綻。
* 記事のタグは人手で設定されていたが、メンテナンスされていなかった。直近の記事ではタグ付けの運用もされておらず、まったくタグのついていない記事や相応しくないタグが設定されていた。
* 無理にSEOを考慮したその場しのぎの設計になっており、ユーザビリティの考慮されていない、拡張性の無い状態。タグ・カテゴリの親子構造もおかしなことになっていた。

##### 担当
* 記事カテゴリの再定義

##### 実績
* 3万件近くあった記事を全てチェックし、既存の記事郡に必要なカテゴリの洗い出し。「1階層8区分」だったカテゴリを「2階層46区分」へ拡張。
* 新しいカテゴリに合わせて既存の記事を適切なカテゴリに修正。アルバイトを1人雇い2人で2ヶ月程度かけ、3万記事のカテゴリ修正を実施。
* 記事の本文からシステムが自動でタグ付けをするよう改修。（ほぼ全ての記事に平均3.75個のタグがつけられた）
* 新しいタグ・カテゴリ構造に合わせアプリケーションの改修 / UIも一新  
  カテゴリ内のテーマの凝集度が高まることによる視認性の向上。子階層へのドリルダウンや関連キーワードといった導線強化による回遊性の向上。
* サイトの世界観にあわないような記事、クォリティの低い記事はビジネスサイドとの協議の上カテゴリごと削除するなど対応を実施。今後そのような記事は作らないよう社内に啓蒙。
* タグ・カテゴリといった記事のメタ情報が細分化されることによって、施策の効果やSEOの推移等ユーザの行動分析等をよりミクロに分析することが可能になった。

##### 技術
* Ruby on Rails
* MySQL
* Elasticsearch

#### 機械学習を用いた記事の自動分類
##### 課題/背景
* 上記サイトリニューアルに伴い、カテゴリが細分化され運用コストが増大
* 複数カテゴリにまたがるような記事も多く人手での判断・分類が困難
* 個々人の判断に委ねるとまた雑多な状態になってしまうリスク

##### 担当
* 教師データのアノテーション
* 機械学習モデルの開発
* 推論API


##### 実績
* 既存の記事をサンプリングし必要なカテゴリの洗い出しを行い、1階層8区分だったカテゴリを2階層30区分へ拡張 
* 記事のカテゴリ分類を行う機械学習モデルの開発
	* カテゴリ毎に特徴語を抽出しLSIで次元削減した特徴量で多クラス分類/ロジスティック回帰 
	* アノテーション用の管理画面を用意し能動学習によって効率的に教師データのラベル付けを行う
    * 推論APIサーバの構築
* 自動タグ付け機能

##### 技術
* Ruby on Rails
* MySQL
* Python (scikit-learn, Gensim etc)
* Elasticsearch

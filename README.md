#### 決済システム内製化
##### 課題/背景
* バンドルカードとはアプリから発行できるVisaプリペイドカード https://vandle.jp/
* サービス開始からカード発行, 決済, 残高管理といったバンドルカードの主要な機能は外部のSaaSを利用していたが、自社でコントロールすることのできない仕様や障害の発生、トランザクションの増加に伴うコスト増が問題となってきていた
* 決済システムを内製化することでコスト削減とシステムの安定化、プロダクトの改善を図る

##### 担当
* カード発行やチャージ, 決済通知等を担うAPIサーバの開発
* Visaの電文を処理するサーバとの間の決済情報の連携
* バンドルカードアプリと通信するAPIサーバ(以下バンドルAPI)におけるSaaS接続部分の内製化への切り替えに伴う改修

##### 実績
* 既存のコードからカードまわりのすべての処理を洗い出しシーケンス図, 状態遷移図の作成
* カード発行やチャージ, 決済通知等を担うAPIサーバの開発
    * バンドルAPIからリクエストを受け以下の処理を行う
        * カード発行, 上位のカードへのアップグレード
        * チャージ, 手数料徴収
        * カードのステータス更新
    * 決済時にバンドルAPIに決済情報を通知
        * Visaの電文を処理するサーバとジョブキューを介して非同期に決済情報を連携し、バンドルAPIに通知を行う
    * OpenAPIによるスキーマ駆動開発
        * バリデーションやルーティング, HTTPリクエスト/レスポンスの構造体などはスキーマから自動生成
* バンドルAPIの改修
    * カードまわりのAPIやバッチ処理を内製化にあわせて改修
    * 分散トランザクションの考慮が甘くリトライ時の多重実行の懸念があった箇所を冪等性を担保できるように改修
    * サーバ同様OpenAPIにてAPIクライアントのコードの自動生成

##### 技術

#### ポチっとチャージ (後払いチャージ)
##### 課題/背景
* ポチっとチャージとはアプリからすぐにチャージができて翌月末までに返済すればよいという後払いサービス
    * それまでのバンドルカードは買い物に使う前にコンビニやペイジーなど事前にチャージが必要であった
    * ちょっとお金が足りなかったり、いますぐ欲しいのにコンビニまで行くのが手間といった問題を解消する機能
* バンドルカード事業の収益の柱として後払いチャージ機能の立ち上げとグロースを担当

##### 担当
* バンドルAPIの後払いチャージ機能開発
* 機械学習モデルの開発
* 推論APIサーバ, データパイプラインの構築
* KPIのモニタリング, 継続的な改善
* 外部の提携企業とのファイル連携/バッチ処理開発

##### 実績
* バンドルAPIの後払いチャージ機能の開発
    * 認証 (メール, SMS)
    * チャージ
    * 支払い (コンビニ, ネット銀行, ペイジー etc)
    * ユーザ毎の申込み可能額の算出 (スコアリング)
    * 機械学習モデルの予測とのインテグレーション
    * 返済期限超過による機能制限
    * メール, SMS, Pushによる督促, IVRによる自動架電システムの構築
    * 複数アカウント等不正検知
* 機械学習モデルの開発
    * サービス開始から1年近くはルールベースの改善によってKPIを伸ばし続けてきたがいくつか課題が出始めた
        1. ルールベースのロジックが複雑化
            * 保守性の低下やバグの温床
        2. ユーザごとの柔軟なスコアリングが困難 / 活用しきれていないデータ
            * ハードコーディングされたロジックでは離散的で画一的な評価になってしまう
            * よりユーザの行動に即した多種多様なデータを反映したいがルールベースでは実装が困難
        3. KPIの成長が鈍化 / 事業の状況に合わせたKPIのコントロールが困難
            * 複数のKPIのトレードオフ(返済率, 売上 etc)を状況に応じて柔軟に調整することが難しい
    * 機械学習で解くべき課題の定義, 評価指標, モデル及び推論APIの構築, バンドルAPIとのインテグレーションまで2ヶ月弱でプロダクションに投入、上記課題に対しては
        1. ルールベースのロジックの複雑化
            * 極力ルールベースから機械学習モデルに判断を寄せることでロジックを簡素化
        2. ユーザごとの柔軟なスコアリングが困難 / 活用しきれていないデータ
            * ルールベースでは扱うことが困難であったデータでも機械学習モデルの特徴量に組み込むことが可能となり画一的でないユーザ毎の評価が可能になった
        3. KPIの成長が鈍化 / 事業の状況に合わせたKPIのコントロールが困難
            * 機械学習モデルを用いる大きなメリットのひとつとして予測を単一の実数値に落とし込むことができる点にあり、意思決定に用いる予測の閾値を調整することで事業の状況にあわせたKPIのコントロールが容易になった
            * 機械学習モデル投入から10ヶ月, サービス開始から2年で当初の事業目標を達成
* 推論APIサーバ, データパイプラインの構築

### iemo株式会社 (2014.08 - 2017.06)
```
住まい/インテリアのメディアでバックエンドや検索, 機械学習を活用した開発に従事
```
#### 検索基盤の整備
##### 課題/背景
* 全文検索にGroongaが使われていたが、ほぼメンテナンスがされておらず冗長化もされていなかった。また仕様を詳しく把握しているエンジニアがいなかった為チューニングや機能追加が困難な状況だった
* 単純なN-gramによる検索だったので検索結果が雑多になりユーザが求める記事に到達することが困難であった
* インデックスの更新がバッチ処理で行われており記事が更新されてから検索に反映されるまでに時間がかかっていた

##### 担当
* 検索エンジンをGroongaからElasticsearchに移行
* 形態素解析の辞書, シノニムの整備
* 検索を利用していたAPIサーバとのインテグレーション
* 全文検索以外へ応用した機能開発

##### 実績
* 検索エンジンをGroongaからElasticsearchに移行し形態素解析を用いたトークナイザの追加, タイトル, 本文, タグ等のウェイトの調整を行い検索後の離脱率の改善 (離脱率: 25.28% -> 15.40% / 検索後記事に辿り着くまでのページ数: 5.25 -> 2.39) 
* 形態素解析の辞書やシノニムの整備し、継続的に誰でもメンテナンスができるように辞書の管理画面を用意
* 記事更新時に非同期にインデックスを更新することで即座に検索に反映
* 全文検索以外へ応用した機能開発
    * 関連記事などのコンテンツベースでのレコメンデーションによる回遊率の改善
    * 形態素解析, TF-IDFを利用した自動タグ付け
    * 記事カテゴリ毎のトピック抽出
    * 禁止ワードの検知

##### 技術
* Elasticsearch
* SQS
* Ruby on Rails
* MySQL


#### サイト構造のリニューアル
##### 課題/背景
* サービス開始から1年以上経過し、記事が雑多になり既存のカテゴリ (インテリア, 家具 etc) の枠組みでは対応できなくなってきていた
    * カテゴリ内の統一性の無さによるユーザビリティの低下, サイトの世界観の破綻
* 記事のタグは人手で設定されていたがメンテナンスされていなかった。直近の記事ではタグ付けの運用もされておらず、まったくタグのついていない記事や相応しくないタグが設定されていた
* 無理にSEOを考慮したその場しのぎの設計になっておりユーザビリティの考慮されていない拡張性の無い状態でカテゴリ/タグの構造も歪になっていた

##### 担当
* ユーザが求めている情報にリーチできるようにすること、その為の課題の洗い出しと改善の実行

##### 実績
* 3万件の記事をチェックし既存の記事郡に必要なカテゴリの洗い出しを実施
    * 「1階層8区分」だったカテゴリを「2階層46区分」へ拡張
* 新しいカテゴリに合わせて記事を適切なカテゴリに修正
    * アルバイトを1人雇い2人で2ヶ月程度かけてすべての記事のカテゴリ修正を実施
* 記事から自動タグ付け (形態素解析, TF-IDF)
* 新しいタグ・カテゴリ構造に合わせてUIをリニューアル
    * カテゴリ内のテーマの凝集度が高まることによる視認性の向上
    * 子階層へのドリルダウンや関連キーワードといった導線強化による回遊性の向上

##### 技術
* Ruby on Rails
* MySQL
* Elasticsearch

#### 機械学習を用いた記事の自動カテゴリ分類
##### 課題/背景
* 上記サイトリニューアルに伴い、カテゴリが細分化され運用コストが増大
* 複数カテゴリにまたがるような記事も多く人手での判断・分類が困難
* 個々人の判断に委ねるとまた雑多な状態になってしまうリスクがあった

##### 担当
* 教師データのアノテーション, 機械学習モデルの開発
* 推論APIサーバの構築
* 記事作成CMSとのインテグレーション


##### 実績
* 記事のカテゴリ分類を行う機械学習モデルの開発
	* カテゴリ毎に特徴語を抽出しLSIで次元削減した特徴量で多クラス分類 (ロジスティック回帰)
	* アノテーション用の管理画面を用意し能動学習によって効率的に教師データのラベル付けを実施
	    * 機械学習モデルの予測確率が曖昧なサンプルを優先的に教師データに追加 (Uncertainty Sampling)
* 推論APIサーバをPythonで構築, 記事作成時にカテゴリを推論

##### 技術
* Ruby on Rails
* MySQL
* Python (scikit-learn, Gensim etc)
* Elasticsearch
